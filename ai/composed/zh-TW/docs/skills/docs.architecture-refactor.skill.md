# Skill: Architecture Documentation Refactor（架構文件重構技能）

## 目的
本技能定義如何將現有程式庫與文件集，轉化為一套結構一致的架構文件。

## 核心原則
架構文件應說明：
- 存在哪些部分
- 為何存在這些部分
- 每個部分的職責是什麼
- 各部分如何相互依賴
- 哪些邊界不得跨越

架構文件不應成為：
- 逐 class 的參考手冊
- API 手冊
- 通用 clean architecture 論文
- 程式碼的複製品

## 審查方法

### A. 先審查文件集
對每份現有文件，判斷：
- 其主要職責是什麼？
- 是否與其他文件重疊？
- 標題是否與實際內容一致？
- 是否包含混合範圍？
- 是否可從 `/docs/README.md` 找到？

常見問題：
- 一份文件包含多個不相關的關注點
- 多份文件以不同方式描述相同邊界
- README 未說明在哪裡找什麼
- 命名反映的是歷史而非當前意義
- 政策文件混雜了規則、實作筆記與遷移待辦事項

### B. 再審查程式庫
從以下推斷架構：
- solution/project 邊界
- folder/module 邊界
- 依賴參照
- composition root / host 設定
- runtime integration 點
- persistence 實作位置
- 抽象介面 vs 實作
- legacy 相容層
- 暗示架構慣例的重複模式

### C. 比對文件與程式碼
將發現分類為：
- 已確認且已記錄
- 已確認但未記錄
- 已記錄但無程式碼佐證
- 不一致
- 不明確

## 文件結構指引

### README 角色
`/docs/README.md` 應包含：
- docs 資料夾的目的
- 文件地圖
- 每份架構文件的簡短說明
- 有用時提供建議閱讀順序
- 目前版本與 legacy 段落的說明（若相關）

### 政策文件角色
每份政策文件理想上應涵蓋一個完整的主題，例如：
- 層次與依賴規則
- project/module 職責
- 組合與 runtime hosting
- 資料存取邊界
- 授權邊界規則
- integration/infrastructure 位置
- legacy 過渡規則

### 應合併的情況
滿足以下條件時合併：
- 兩份文件太小且強相關
- 一份文件的存在只因為舊命名
- 多份文件重複相同規則

### 應拆分的情況
滿足以下條件時拆分：
- 一份文件混合了不相關的主題
- 一份文件太大而難以導覽
- 明顯存在不同的受眾或維護節奏

### 應重新命名的情況
滿足以下條件時重新命名：
- 目前名稱模糊
- 目前名稱反映的是實作歷史而非架構意義
- 目前名稱導致對職責或範圍的誤解

## 撰寫規則
對每個架構主題，偏好以下結構：

### 目的
此部分/文件的存在理由。

### 範圍
包含什麼。

### 職責
必須擁有什麼。

### 非職責 / 邊界
不得做什麼。

### 依賴方向
可依賴什麼，以及什麼可依賴它。

### 備註
例外、遷移說明或 legacy 注意事項。

## 佐證規則
可接受的佐證：
- 多個一致的程式碼模式
- project 參照
- DI/composition 設定
- 與程式碼一致的現有文件
- 足夠強烈的命名與使用情境

薄弱的佐證：
- 單一孤立的 class 名稱
- 推測性的資料夾命名
- 與實際依賴使用相矛盾的 comment

當佐證薄弱時，標記為：
- `Assumption`
- `To be confirmed`

## 最終輸出預期
重構文件後：
- `/docs/README.md` 必須準確描述文件結構
- 每份政策文件應有明確的目的
- 重複的架構說明應最小化
- 未解決的不一致應明確列出
